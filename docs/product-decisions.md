# Stock Insight 설계 의사결정 문서

## 1. 문서 목적

이 문서는 [requirements.md](C:/Users/123/개발/stock-insight/docs/requirements.md)를 변경하지 않고, 그 요구사항을 실제 설계로 옮기기 위한 정책 결정을 정리한다.

이 문서의 역할은 다음과 같다.

- DB 설계 전에 핵심 정책을 고정한다.
- API와 데이터 모델 설계의 기준을 만든다.
- 구현 단계에서 모호한 해석을 줄인다.

`requirements.md`가 "무엇을 만들 것인가"를 정의한다면, 이 문서는 "그것을 어떤 방식으로 만들 것인가"를 정의한다.

## 2. 현재 설계 범위

이번 설계는 MVP 범위만 다룬다.

- 대상: 일반 국내 주식
- 우선 시장: `KOSPI`, `KOSDAQ`
- 데이터 축: 종목, 일봉 가격, 뉴스, 감성 분석, AI 인사이트, 인사이트 검증

이번 문서에서 제외하는 내용:

- 해외 주식 확장 상세 설계
- 유튜브 수집/분석 상세 설계
- 사용자 계정 및 권한
- 운영 대시보드 UI 상세 설계

## 3. 제품 해석 원칙

### 3.1 이 서비스는 "판단 보조 도구"다

서비스는 매수/매도 추천 엔진이 아니라, 사용자의 해석을 돕는 리서치 보조 도구로 설계한다.

따라서 모든 인사이트는:

- 최종 결론만 보여주지 않고
- 근거 데이터와 함께 제시되며
- 나중에 검증 가능한 기록으로 남아야 한다.

### 3.2 종목 중심 조회와 이슈 기반 탐색을 함께 설계한다

MVP에서는 사용자가 "특정 종목을 조회"하는 흐름과, "최근 이슈를 바탕으로 관심 종목을 발견"하는 흐름을 함께 고려한다.

핵심 진입점:

- 종목 검색
- 종목 상세 페이지
- 이슈/뉴스 기반 관심 종목 탐색
- 종목별 가격/뉴스/인사이트 조회

### 3.3 설명 가능성을 정확도보다 우선한다

MVP에서는 모델의 예측 정확도를 높이는 것보다, 왜 그런 해석이 나왔는지를 이해할 수 있게 만드는 것을 우선한다.

## 4. MVP 정책 결정

### 4.1 종목 범위

결정:

- MVP 대상은 일반 국내 상장 주식으로 한정한다.
- 시장 범위는 `KOSPI`, `KOSDAQ`를 포함한다.
- `KONEX`, ETF, ETN, 리츠, 채권형 상품은 MVP에서 제외한다.

이유:

- 국내 주식 MVP라는 요구사항에 가장 잘 맞는다.
- 검색, 가격 수집, 뉴스 연결 품질을 안정적으로 관리하기 쉽다.
- 자산 유형이 섞이면 인사이트 해석 기준이 흔들릴 수 있다.

설계 영향:

- `stocks`에는 시장 구분 필드가 필요하다.
- 종목 마스터 수집 단계에서 자산 유형 필터링 정책이 필요하다.

### 4.2 가격 데이터 범위

결정:

- MVP 가격 데이터는 일봉만 저장한다.
- 저장 단위는 거래일 기준 1건이다.
- 기준 시세는 종가를 기본 참조값으로 사용한다.

이유:

- 프로젝트의 핵심은 초단타 대응이 아니라 흐름 해석과 사후 검증이다.
- 인사이트 검증도 일봉 종가 기준이면 충분히 시작할 수 있다.

설계 영향:

- `stock_price_daily`는 종목 + 거래일 unique 제약이 필요하다.
- 인사이트 평가도 기준 가격과 평가 가격을 일봉 종가 기준으로 계산한다.

### 4.3 뉴스 저장 정책

결정:

- 뉴스는 제목, 요약 또는 정제 본문, URL, 출처, 발행 시각을 저장한다.
- 원문 전체 저장은 옵션으로 두되, MVP 기본 정책은 "요약 또는 정제 본문 우선"으로 한다.
- 원문 확인은 URL 링크를 통해 사용자가 직접 확인할 수 있게 한다.

이유:

- 원문 전체 저장은 저작권/중복/정제 품질 이슈가 있다.
- MVP에서는 분석과 근거 제시에 필요한 최소 저장 구조가 더 중요하다.

설계 영향:

- `news_articles`에는 `title`, `summary`, `body_text`, `source_name`, `url`, `published_at`가 후보가 된다.
- `body_text`는 nullable로 두고, 소스별로 확보 가능한 경우만 저장한다.

### 4.4 뉴스와 종목의 관계

결정:

- 뉴스와 종목의 관계는 N:M으로 설계한다.

이유:

- 하나의 뉴스가 여러 종목에 영향을 줄 수 있다.
- 테마주/섹터/공급망 관련 기사에서 단일 종목 매핑으로는 정보 손실이 크다.

설계 영향:

- 조인 테이블 `news_article_stocks`가 필요하다.
- 연결의 생성 방식은 초기에는 규칙 기반 또는 검색 기반으로 단순하게 시작한다.

### 4.5 감성 분석 정책

결정:

- 감성 분석 결과는 뉴스 기사 단위로 저장한다.
- 결과는 `positive`, `neutral`, `negative` 3단계로 시작한다.
- 감성 결과는 덮어쓰기하지 않고 실행 이력을 남기는 구조를 우선한다.

이유:

- 같은 기사도 프롬프트/모델 변경에 따라 결과가 달라질 수 있다.
- 나중에 품질 비교나 재분석을 위해 실행 이력이 유용하다.

설계 영향:

- `news_sentiment_runs`는 기사와 1:N 관계를 가진다.
- 최신 결과 조회 로직과 전체 이력 조회 로직을 분리할 수 있어야 한다.

### 4.6 AI 인사이트 생성 정책

결정:

- 인사이트는 종목 기준으로 생성한다.
- 인사이트는 스냅샷으로 저장한다.
- 생성 시점의 입력 범위와 핵심 근거를 함께 저장한다.
- 최신 인사이트와 과거 인사이트를 모두 보존한다.

이유:

- 이 프로젝트의 중요한 가치 중 하나는 사후 검증이다.
- 최신 결과만 남기면 과거 해석을 평가할 수 없다.

설계 영향:

- `stock_insights`는 append-only 성격을 가진다.
- 인사이트에는 입력 데이터 범위, 생성 시점, 모델 정보가 포함되어야 한다.

### 4.7 인사이트 검증 정책

결정:

- 인사이트 생성 시점의 기준 종가를 저장한다.
- 검증 구간은 최소 `7일`, `30일`, `90일`, `180일`로 고정한다.
- 각 구간별 실제 종가와 수익률 변화를 기록한다.
- MVP에서는 "상승/하락/중립 방향성" 기준 평가를 우선 지원한다.

이유:

- 장기 검증 목적에 가장 직접적으로 연결된다.
- 수익률 수치뿐 아니라 방향성 적중 여부가 사용자의 학습에 중요하다.

설계 영향:

- `stock_insight_evaluations`는 인사이트 1건당 여러 평가 시점을 가질 수 있다.
- 평가 배치가 재실행될 수 있으므로 unique 정책이 필요하다.

### 4.8 유튜브/시황 콘텐츠 정책

결정:

- 유튜브 및 외부 시황 콘텐츠는 MVP에서 제외한다.
- 다만 향후 확장을 고려해 "외부 콘텐츠"를 수용할 수 있는 느슨한 구조 가능성은 열어둔다.

이유:

- 자막/대본 수집 안정성, 정제 품질, 종목 매핑 난이도가 높다.
- 현재 핵심 가치인 종목-뉴스-인사이트-검증 플로우를 먼저 완성하는 것이 낫다.

설계 영향:

- MVP 스키마에는 유튜브 테이블을 넣지 않는다.
- 향후 `content_sources` 계열 확장 문서에서 별도 설계한다.

### 4.9 관심 종목 탐색 정책

결정:

- 최근 뉴스/이슈를 바탕으로 관심 종목 후보를 제안하는 기능을 요구사항 범위에 포함한다.
- 이 기능은 투자 추천이 아니라 탐색 보조 기능으로 설계한다.
- 제안 결과는 반드시 근거와 함께 제공한다.

이유:

- 사용자는 어떤 종목을 봐야 할지 모르는 상태에서 탐색을 시작할 필요가 있다.
- 프로젝트의 가치가 "조회"를 넘어 "탐색 시작점 제공"까지 확장된다.

설계 영향:

- 종목 검색 외에 탐색용 API 또는 화면이 필요하다.
- 추천 근거를 설명할 수 있는 데이터 조합 로직이 필요하다.
- 초기에는 별도 테이블보다 집계/계산 기반 조회로 시작하는 편이 적절하다.

## 5. 핵심 도메인 정의

### 5.1 Stock

국내 상장 주식 종목의 기준 마스터 데이터.

역할:

- 종목 검색의 기준이 된다.
- 가격, 뉴스, 인사이트의 연결 축이 된다.

### 5.2 DailyPrice

종목별 거래일 기준 가격 데이터.

역할:

- 차트 조회의 기반이 된다.
- 인사이트 생성과 검증의 기준 데이터가 된다.

### 5.3 NewsArticle

종목과 연결되는 외부 뉴스 콘텐츠.

역할:

- 최근 이슈 흐름을 보여준다.
- 감성 분석과 인사이트의 입력 데이터가 된다.

### 5.4 NewsSentimentRun

특정 뉴스 기사에 대해 실행된 감성 분석 결과.

역할:

- 분석 결과의 이력을 남긴다.
- 최신 감성 상태를 조회할 수 있게 한다.

### 5.5 StockInsight

특정 시점에 특정 종목에 대해 생성된 AI 해석 결과.

역할:

- 종목에 대한 요약 해석을 제공한다.
- 이후 검증 대상이 되는 스냅샷 역할을 한다.

### 5.6 StockInsightEvaluation

과거 인사이트가 이후 가격 흐름과 얼마나 부합했는지 평가한 결과.

역할:

- 사후 검증 기능의 중심이 된다.
- 인사이트 신뢰도 측정의 기반이 된다.

### 5.7 CollectionJob / CollectionRun

가격/뉴스/분석 작업의 실행 단위와 실행 이력.

역할:

- 운영 추적성을 확보한다.
- 수집 실패와 재실행 이력을 관리한다.

## 6. 도메인 관계 초안

현재 관계 초안은 아래와 같다.

- `Stock` 1:N `DailyPrice`
- `Stock` N:M `NewsArticle`
- `NewsArticle` 1:N `NewsSentimentRun`
- `Stock` 1:N `StockInsight`
- `StockInsight` 1:N `StockInsightEvaluation`
- `CollectionJob` 1:N `CollectionRun`

추가 고려:

- `CollectionRun`이 어떤 엔티티를 대상으로 실행되었는지 추적할지 여부
- `StockInsight`와 사용된 `NewsArticle` 사이의 명시적 관계를 남길지 여부
- 관심 종목 제안 결과를 영속 저장할지, 조회 시점 계산으로 생성할지 여부

이 부분은 ERD 단계에서 확정한다.

## 7. API 설계에 반영할 기본 원칙

### 7.1 종목 상세 응답은 "탭별 조합"을 고려한다

종목 상세 화면은 한 번에 모든 데이터를 거대 응답으로 주기보다, 아래처럼 분리 가능한 형태를 선호한다.

- 기본 정보
- 가격 데이터
- 뉴스 목록
- 최신 인사이트
- 인사이트 평가

### 7.2 최신값과 이력을 구분한다

감성 분석과 인사이트는 모두 "최신 조회"와 "이력 조회"를 분리해서 설계하는 것이 좋다.

### 7.3 검증 API는 독립 엔드포인트로 둔다

인사이트 검증은 대시보드의 부가 기능이 아니라 핵심 가치이므로 별도 조회 단위를 두는 편이 좋다.

## 8. 파이프라인 설계에 반영할 기본 원칙

### 8.1 수집과 분석을 분리한다

추천 흐름:

1. 종목 마스터 수집
2. 일봉 가격 수집
3. 뉴스 수집
4. 뉴스 감성 분석
5. 종목 인사이트 생성
6. 인사이트 평가 배치 실행

### 8.2 분석은 재실행 가능해야 한다

감성 분석과 인사이트 생성은 모두 모델 변경, 프롬프트 개선, 실패 복구를 고려해 재실행 가능해야 한다.

### 8.3 평가 배치는 지연 실행 구조가 필요하다

`7일`, `30일`, `90일`, `180일` 평가 시점은 인사이트 생성 후 미래 시점에 실행되어야 하므로, 지연 실행 또는 주기 배치 구조를 고려한다.

## 9. 남아 있는 설계 쟁점

아래 항목은 다음 문서에서 더 구체화해야 한다.

- 종목 메타데이터 필드 범위
- 뉴스 종목 연결 방식
- 인사이트 본문 구조화 수준
- 평가 결과를 단순 수익률로 볼지, 방향성 레이블까지 함께 둘지
- 인사이트 생성 시 사용한 뉴스 목록을 명시적으로 저장할지 여부
- 이슈/테마 기반 종목 추천 기능을 MVP 이후 어떤 방식으로 도입할지 여부
