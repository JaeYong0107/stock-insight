# Stock Insight 요구사항 정의서

## 1. 문서 목적

이 문서는 `Stock Insight` 프로젝트의 제품 방향, MVP 범위, 요구사항, 구현 우선순위를 정의한다.

이 문서를 기준으로:

- 무엇을 만들지 먼저 합의한다.
- DB 설계 전에 필요한 정책을 결정한다.
- 구현은 체크리스트를 따라 순차적으로 진행한다.
- 요구사항 변경 시 코드보다 문서를 먼저 수정한다.

---

## 2. 프로젝트 한 줄 정의

국내 주식 투자자가 종목 데이터, 뉴스, 해석 근거, AI 인사이트를 한곳에서 보고 스스로 판단할 수 있도록 돕는 개인용 주식 리서치 비서를 만든다.

---

## 3. 이 프로젝트를 만드는 이유

프로젝트의 출발점은 "직접 인사이트를 도출하는 훈련을 돕는 도구"가 필요하다는 문제의식이다.

사용자는 재테크에 관심을 가지게 되면서 장기 투자 상품을 모으고 있지만, 개별 주식으로 관심 범위를 넓히려면 아래 과정이 필요하다고 느끼고 있다.

- 뉴스 읽기
- 관심 산업과 기업에 대한 검색
- 시황/시사 콘텐츠 구독
- 여러 근거를 종합해 스스로 해석하기

하지만 처음에는 이 과정을 매번 직접 수행하는 진입장벽이 높다. 그래서 이 프로젝트는 투자 판단을 대신 내리는 서비스가 아니라, 사용자가 판단할 수 있도록 근거를 구조화해 주는 "나만의 주식 비서"를 목표로 한다.

핵심 철학은 다음과 같다.

- AI가 결론을 대신 내리지 않는다.
- 데이터와 근거를 먼저 제시한다.
- 사용자가 그 해석이 타당한지 직접 판단한다.
- 남겨진 기록으로 나중에 해석의 품질과 신뢰도를 검증한다.

---

## 4. 제품 비전

이 프로젝트는 단순한 주가 조회 서비스가 아니라 다음 역할을 수행해야 한다.

- 특정 국내 종목의 최근 흐름을 빠르게 요약한다.
- 관련 뉴스와 데이터 근거를 구조화해 보여준다.
- AI가 해석을 제안하되, 근거와 함께 제시한다.
- 시간이 지난 뒤 그 해석이 실제로 맞았는지 추적한다.

장기적으로는 사용자가 반복적으로 이 도구를 사용하면서:

- 시장과 종목을 보는 프레임을 익히고
- 자신만의 판단 기준을 만들고
- 어떤 해석이 실제로 유효했는지 검증하는 경험을 쌓는 것을 목표로 한다.

---

## 5. 목표 사용자

### 5.1 1차 대상

- 국내 주식에 관심이 있는 개인 투자자
- 기업/뉴스/가격 흐름을 함께 보고 싶은 입문자
- 인사이트를 자동으로 받기보다, 판단 근거를 보고 스스로 검토하고 싶은 사용자

### 5.2 사용자 특성

- 매매 실행 기능보다 리서치 보조 기능이 더 중요하다.
- 정답 예측보다 "왜 이런 해석이 나왔는지"를 더 중요하게 본다.
- 시간이 지나서도 과거 해석을 다시 보고 검증하고 싶어 한다.

---

## 6. MVP 범위

MVP는 일반 국내 주식만 대상으로 한다.

### 포함

- 국내 상장 주식 종목 마스터 관리
- 종목 검색
- 뉴스/이슈 기반 관심 종목 탐색
- 종목 기본 정보 조회
- 일봉 주가 데이터 수집 및 조회
- 종목 관련 뉴스 수집 및 조회
- 뉴스 기반 감성 분석
- 종목별 AI 인사이트 생성
- 인사이트의 근거 데이터 요약 제공
- 종목 상세 대시보드 제공
- 생성된 인사이트와 이후 실제 가격 흐름을 비교하기 위한 기록 구조

### 제외

- 해외 주식
- ETF, 채권, 코인, 파생상품
- 실시간 체결/호가 수준의 초단타 데이터
- 실제 매매 기능
- 사용자 로그인 및 결제
- 알림 발송
- 모바일 앱
- 유튜브 자동 수집 및 분석

### 참고

유튜브 스크립트 기반 시황 분석은 아이디어 가치가 높지만, 데이터 수집 난이도와 품질 관리 이슈가 있어 MVP 이후 확장 항목으로 둔다.

---

## 7. 제품 원칙

### 7.1 판단 보조 중심

서비스는 투자 추천 엔진이 아니라 판단 보조 도구여야 한다.

### 7.2 근거 우선

AI 요약보다 먼저 데이터, 뉴스, 근거 포인트가 보여야 한다.

### 7.3 추적 가능성

모든 인사이트는 언제, 어떤 데이터로 생성되었는지 추적 가능해야 한다.

### 7.4 사후 검증 가능성

과거에 제시한 해석과 이후 실제 결과를 비교할 수 있어야 한다.

### 7.5 국내 주식 MVP 집중

초기에는 범위를 좁혀서 완성도를 높이고, 이후 해외 주식과 추가 데이터 소스로 확장한다.

### 7.6 탐색 시작점 제공

서비스는 사용자가 이미 알고 있는 종목만 조회하는 도구에 머물지 않아야 한다.

최근 뉴스와 이슈를 바탕으로 "관심 가져볼 만한 종목"을 제안해, 사용자가 탐색을 시작할 수 있는 진입점을 제공해야 한다.

---

## 8. 해결하려는 핵심 문제

사용자는 특정 종목을 공부하려고 할 때 정보를 너무 많은 채널에서 수집해야 한다.

- 주가 흐름은 MTS/포털/증권사 앱에서 본다.
- 뉴스는 포털과 검색에서 따로 찾아야 한다.
- 시황 해설은 유튜브나 블로그에서 따로 봐야 한다.
- 그 모든 정보를 종합해서 스스로 해석해야 한다.

이 프로젝트는 그 부담을 줄이기 위해 다음을 제공해야 한다.

- 종목 기준으로 정보 모으기
- 최근 변화 포인트 요약하기
- 해석의 근거를 구조화하기
- 시간이 지난 뒤 해석 품질을 검증하기

---

## 9. 핵심 사용자 시나리오

### 시나리오 1. 종목 탐색

사용자는 종목명 또는 종목코드로 국내 상장 종목을 검색할 수 있어야 한다.

### 시나리오 1-1. 이슈 기반 탐색

사용자는 최근 뉴스나 시장 이슈를 바탕으로, 추가로 살펴볼 만한 종목 후보를 제안받을 수 있어야 한다.

이 제안은 투자 추천이 아니라 탐색 보조여야 하며, 왜 그 종목이 제안되었는지 근거를 함께 확인할 수 있어야 한다.

### 시나리오 2. 종목 현황 파악

사용자는 특정 종목의 최근 일봉 흐름, 등락률, 거래량 변화 등 기본 흐름을 한눈에 볼 수 있어야 한다.

### 시나리오 3. 뉴스 확인

사용자는 해당 종목과 관련된 최신 뉴스 목록과 주요 내용을 확인할 수 있어야 한다.

### 시나리오 4. 해석 근거 확인

사용자는 단순 감성 결과뿐 아니라, 어떤 뉴스와 어떤 데이터가 해석 근거가 되었는지 확인할 수 있어야 한다.

### 시나리오 5. AI 인사이트 검토

사용자는 AI가 생성한 종합 코멘트를 읽고, 그 해석이 타당한지 스스로 판단할 수 있어야 한다.

### 시나리오 6. 사후 검증

사용자는 과거 특정 시점의 인사이트를 다시 보고, 이후 1주, 1개월, 3개월, 6개월 등의 가격 흐름과 비교해 방향성이 맞았는지 검토할 수 있어야 한다.

## 10. MVP 핵심 기능

### 10.1 종목 검색 및 상세 조회

- 국내 주식 종목 검색
- 최근 뉴스/이슈 기반 관심 종목 제안
- 종목 기본 정보 제공
- 종목 상세 페이지 진입

### 10.2 주가 데이터 조회

- 일봉 기준 가격 데이터 조회
- 최근 기간별 흐름 확인
- 거래량 및 변동률 확인

### 10.3 뉴스 수집 및 조회

- 종목 관련 뉴스 수집
- 뉴스 목록 조회
- 뉴스 원문 링크 제공
- 뉴스 발행 시각과 출처 표시

### 10.4 뉴스 감성 분석

- 뉴스별 긍정/중립/부정 분류
- 간단한 근거 또는 요약 코멘트 제공

### 10.5 종목 인사이트 생성

- 최근 가격 흐름과 뉴스를 바탕으로 인사이트 생성
- 핵심 근거 포인트 포함
- 생성 시점과 입력 범위 기록

### 10.6 인사이트 검증 기록

- 인사이트 생성 시점의 기준 가격 저장
- 이후 특정 시점의 실제 가격과 비교 가능하도록 기록 구조 제공

## 11. 기능 요구사항 체크리스트

### 11.1 종목 도메인

- [ ] 국내 상장 일반 주식을 식별할 수 있어야 한다.
- [ ] 종목은 종목코드, 종목명, 시장 구분 정보를 가져야 한다.
- [ ] 종목 검색은 이름/코드 기준으로 동작해야 한다.
- [ ] 상장폐지 또는 비활성 종목 상태를 구분할 수 있어야 한다.
- [ ] 동일 종목이 중복 저장되지 않아야 한다.

### 11.1-A 탐색 제안

- [ ] 최근 뉴스와 이슈를 기반으로 관심 종목 후보를 제안할 수 있어야 한다.
- [ ] 제안 기능은 "매수 추천"이 아니라 "탐색 후보 제시"로 동작해야 한다.
- [ ] 사용자는 왜 해당 종목이 제안되었는지 근거를 확인할 수 있어야 한다.
- [ ] 제안 근거에는 최소 뉴스 언급량, 관련 키워드 또는 이슈, 최근 가격 변화 중 일부가 포함되어야 한다.
- [ ] 사용자는 제안된 종목에서 바로 상세 화면으로 이동할 수 있어야 한다.

### 11.2 주가 데이터

- [ ] 종목별 일봉 데이터를 저장할 수 있어야 한다.
- [ ] 가격 데이터는 일자, 시가, 고가, 저가, 종가, 거래량을 포함해야 한다.
- [ ] 특정 종목의 최근 가격 데이터를 기간별로 조회할 수 있어야 한다.
- [ ] 특정 기준일의 종가를 쉽게 조회할 수 있어야 한다.
- [ ] 데이터 수집 시 동일 종목/동일 일자의 가격 데이터가 중복 저장되지 않아야 한다.
- [ ] 추후 분봉 또는 실시간 확장을 고려할 수 있는 구조여야 한다.

### 11.3 뉴스 데이터

- [ ] 종목 관련 뉴스를 수집할 수 있어야 한다.
- [ ] 뉴스는 제목, 요약 또는 본문, 출처, URL, 발행 시각을 가져야 한다.
- [ ] 하나의 뉴스가 여러 종목과 연결될 수 있어야 한다.
- [ ] 동일 뉴스 URL 또는 동일 원문이 중복 저장되지 않아야 한다.
- [ ] 최신순으로 뉴스 조회가 가능해야 한다.
- [ ] 뉴스 원문 링크를 통해 사용자가 직접 출처를 확인할 수 있어야 한다.

### 11.4 감성 분석

- [ ] 뉴스 단위 감성 분석 결과를 저장할 수 있어야 한다.
- [ ] 감성 값은 최소 `positive`, `neutral`, `negative` 중 하나여야 한다.
- [ ] 감성 분석 결과에는 짧은 설명 또는 근거 코멘트를 포함할 수 있어야 한다.
- [ ] 감성 분석 실행 시점과 사용 모델 정보를 기록할 수 있어야 한다.
- [ ] 감성 분석 결과는 재실행을 고려해 이력 관리가 가능해야 한다.

### 11.5 AI 인사이트

- [ ] 종목 기준 종합 인사이트를 생성할 수 있어야 한다.
- [ ] 인사이트는 생성 시점이 기록되어야 한다.
- [ ] 인사이트는 어떤 데이터 범위를 기반으로 생성되었는지 추적 가능해야 한다.
- [ ] 인사이트는 핵심 주장뿐 아니라 근거 요약을 포함해야 한다.
- [ ] 인사이트는 종목 상세 화면에서 최신 결과를 우선 보여줘야 한다.
- [ ] 인사이트는 향후 버전 관리 또는 재생성을 고려할 수 있어야 한다.

### 11.6 인사이트 검증

- [ ] 인사이트 생성 당시 기준 가격을 저장할 수 있어야 한다.
- [ ] 이후 시점의 실제 가격과 비교할 수 있어야 한다.
- [ ] 최소 1주, 1개월, 3개월, 6개월 기준의 성과 검증을 지원할 수 있어야 한다.
- [ ] 해석이 완전히 맞았는지뿐 아니라 방향성 적중 여부도 판단할 수 있어야 한다.
- [ ] 검증 로직은 나중에 고도화할 수 있도록 독립된 구조를 가져야 한다.

### 11.7 대시보드 조회

- [ ] 종목 상세 화면에서 가격, 뉴스, 감성, 인사이트를 함께 조회할 수 있어야 한다.
- [ ] 사용자는 인사이트와 함께 근거 데이터를 확인할 수 있어야 한다.
- [ ] 프론트엔드는 백엔드 API를 통해 필요한 데이터를 조합해 표시할 수 있어야 한다.
- [ ] 조회 응답은 사용자 체감상 빠르게 로드되어야 한다.

### 11.7-A 탐색 화면 조회

- [ ] 사용자는 홈 화면 또는 별도 탐색 화면에서 최근 이슈 기반 관심 종목 후보를 조회할 수 있어야 한다.
- [ ] 각 후보에는 종목명, 제안 근거 요약, 관련 뉴스 수 또는 관련 키워드가 포함되어야 한다.
- [ ] 제안 결과는 설명 가능해야 하며 블랙박스 점수만 노출하지 않아야 한다.

### 11.8 데이터 수집 작업

- [ ] 주가 수집 작업은 정기 실행할 수 있어야 한다.
- [ ] 뉴스 수집 작업은 정기 실행할 수 있어야 한다.
- [ ] 수집 실패 시 실패 원인과 재시도 여부를 기록할 수 있어야 한다.
- [ ] 마지막 성공 시각을 확인할 수 있어야 한다.
- [ ] 소스별 수집 상태를 추적할 수 있어야 한다.

## 12. 비기능 요구사항

### 12.1 신뢰성

- [ ] 동일 데이터 중복 저장 방지 규칙이 있어야 한다.
- [ ] 수집 원본과 생성 원본을 추적할 수 있어야 한다.
- [ ] 인사이트 생성 시 사용한 입력 데이터 시점을 확인할 수 있어야 한다.

### 12.2 설명 가능성

- [ ] 사용자가 "왜 이런 결론이 나왔는지" 이해할 수 있어야 한다.
- [ ] 블랙박스 형태의 최종 결론만 노출하지 않아야 한다.
- [ ] 관심 종목 제안 역시 왜 제안되었는지 근거를 설명할 수 있어야 한다.

### 12.3 확장성

- [ ] 국내 주식 MVP 이후 해외 주식으로 확장 가능해야 한다.
- [ ] 뉴스 소스가 추가되어도 구조 변경이 최소화되어야 한다.
- [ ] AI 모델이 바뀌어도 저장 구조가 유지되어야 한다.
- [ ] 향후 유튜브 스크립트/시황 콘텐츠 분석이 추가 가능해야 한다.

### 12.4 운영성

- [ ] 수집 작업 상태를 확인할 수 있어야 한다.
- [ ] 장애 발생 시 원인을 추적할 수 있는 로그가 필요하다.
- [ ] 환경 변수 기반 설정이 가능해야 한다.

## 16. DB 설계 전에 확정해야 할 핵심 정책

- [x] MVP 대상은 일반 국내 주식으로 한정한다.
- [ ] 국내 주식 범위에서 우선 시장을 KOSPI/KOSDAQ 모두 포함할지 확정한다.
- [ ] 종목 기본 정보에 어떤 메타데이터까지 저장할지 확정한다.
- [ ] 가격 데이터는 MVP에서 일봉만 저장한다.
- [ ] 뉴스는 본문 전체 저장과 요약 저장 중 어떤 정책을 기본으로 할지 확정한다.
- [ ] 뉴스와 종목의 관계는 N:M으로 설계한다.
- [ ] 감성 분석 결과는 덮어쓰기보다 이력 보존 가능 구조를 우선 검토한다.
- [ ] AI 인사이트는 스냅샷으로 저장한다.
- [ ] 인사이트 검증용 기준 시점과 검증 시점 정책을 정의한다.
- [ ] 유튜브 데이터는 백로그로 미루고, 스키마는 느슨한 확장 가능성만 남긴다.
